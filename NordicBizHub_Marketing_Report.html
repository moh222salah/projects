الحل #1: إعادة بناء GoogleController بنظام Stateless Authentication — صفر أخطاء

الحل #2: استعادة كاملة لدورة حياة الطلبات من أول خطوة لآخر خطوة
اليوم 1–2 — تحليل Root Cause
اكتشاف السبب الجذري للأعطال
السبب: تضارب Session بين النطاقات الفرعية. Laravel Session غير متوافق مع OAuth Callback domain — CSRF tokens تُبطل قبل استجابة Google مما يولد خطأ 419 في كل محاولة تسجيل دخول.
اليوم 3–4 — Stateless Auth Rebuild
إعادة بناء GoogleController من الصفر
حذف الاعتماد الكامل على Session في OAuth. توليد تلقائي لكلمة المرور (bcrypt). ربط الأدوار تلقائياً post-login. تأمين redirect URI. النتيجة النهائية: صفر أخطاء 419 على جميع المتصفحات.
اليوم 5–7 — Flow Restoration
استعادة Service Request Flow الكامل
إصلاح Combined Order Pipeline: اختيار خدمات ← إجابة أسئلة ← مراجعة ← تقديم. التحقق من جميع Middleware الأدوار. اختبار شامل لـ Admin / Employee / Client. المنصة جاهزة للإنتاج.

Retry Logic — 3 محاولات
Exponential Backoff (30s → 60s). إذا فشلت جميع المحاولات يُسجَّل الخطأ مع User ID وError Message كاملة بدون أي تأثير على تجربة المستخدم.
⏱
Timeout Protection
SyncCustomerToERP: 60 ثانية. SyncOrderToERP: 90 ثانية (Invoice + Project متوازيان). يمنع العمليات المعلقة من تعطيل Queue Worker.
🗄
Database Queue — Persistent
QUEUE_CONNECTION=database. الوظائف محفوظة في قاعدة البيانات وتبقى بعد أي restart للسيرفر. Daemon Worker يعمل بشكل مستمر.
🔕
Zero UI Impact — مضمون
تجربة المستخدم لا تُلمس. جميع عمليات ERP في الخلفية الكاملة. المستخدم يُسجَّل وينتقل فوراً — لا انتظار لأي API Response.
